# contracts.md — WorkOS numbered invariants

Skills and docs reference these **by number + name** (e.g. "per C7 salesforce-read-only-and-
optional") and never restate their content — CI greps for restatement. Changing a contract is
a reviewed PR that names every consumer it touches. Numbers were assigned at Phase 0A
(2026-07-15) and are stable; add new contracts at the end, never renumber.

**C1 · engine-memory-split** — Skills, templates, artifact shells, contracts live in this
repo. Customer/account data lives only in tenant-controlled surfaces (each person's memory
folder; the shared `Team/` publication folder) — never in this repo, and never in its git
history. Ports copy files, never history. All examples in repo files are fictional.

**C2 · identity-config** — Every personal value (memory-root path AND name, user name,
initials, manager, capability tiers, integrations, fiscal offset, publish folder) resolves
through the per-person config defined in `shared/identity.schema.md`, generated by `setup`'s
question flow — never hand-edited, never hardcoded in a skill. A skill containing a literal
personal value fails review.

**C3 · time-agnostic-passes** — No pass ever reasons about what time of day it is. "Close
whatever is open, flush, open today's" must work at 4am, noon, or twice on a Tuesday.
Cadence lives in scheduled-task configuration, not in skill logic.

**C4 · single-writer-state** — Exactly one pass writes `state/` at a time. A starting pass
checks for a running sibling; the locking mechanism and stale-lock recovery follow the
Phase 0 spike's recorded design. `state/` files have exactly one writer per run; artifacts
and other skills read, never write.

**C5 · draft-before-write** — A human approves content before it is written, published, or
staged for an external surface — shown the actual artifact, not a description of it.
**Exemption:** regenerable, dated report artifacts that are re-run live each time may save
directly; anything writing account truth, editing existing content, or leaving the tenant
keeps the gate. Approval happens once per artifact — callers of a delegate present the
delegate's output at their own single gate (no double-gating), and the approving side is the
one writer of any resulting log entry.
**State-store clause (added 2026-07-15, sync review):** machine-written operational state
under `state/` is bookkeeping, not a user artifact — calendar/harvest-derived updates and
newly-formed additive entries may be written ungated; user-meaningful mutations (task
closures, deletions, rewrites, unsuppressions, file moves/deletions) queue as
`pendingApprovals` and apply only inside an attended run's approval step (consolidated
display; destructive items decided per item — C14).
Append-only journal POINTER lines are exempt bookkeeping, as is the engine version beacon
(`Team/_engine/latest-version.txt` — monotonic bump only, per `shared/version-check.md`;
added 2026-07-17, #29).

**C6 · no-shadow-store** — Never store a copy of an authoritative source's current state,
and never store a derived value next to its source (scores/signals compute at render time).
**Identity clause:** stable identifiers (e.g. a Salesforce Account Id) are identity, not
deal-state — cacheable with provenance (`source/confidence/last_verified`) and an
invalidation rule. **next-step-history clause:** an append-only dated record of observations
and approved outputs is a history, not a shadow — permitted, but never read back as current
state.

**C7 · salesforce-read-only-and-optional** — No write path to Salesforce exists at any tier;
outputs are paste-ready and never claimed as written. Salesforce access is OPTIONAL:
capability comes from C2's `sfdc_tier`; at the `manual` tier, pasted reports/screenshots are
first-class authoritative intake and `01_Opportunities/{OppNumber}_{Label}/` folder names are
the local opportunity registry. A skill that hard-fails without Salesforce fails review.

**C8 · skills-bundle-their-assets** — A skill may only read its own bundle at runtime —
never repo paths, never another skill. Shared content (`shared/`) is injected into each
bundle's `assets/shared/` at build/stage time; injected copies are build artifacts
(gitignored), and `.skill` bundles themselves are untracked build artifacts. No runtime
skill-chaining dependencies: delegate patterns exchange text within a session, not installs.

**C9 · artifact-rebuild** — A pass rewriting an artifact replaces ONLY its marked data
blocks and preserves all surrounding markup/JS verbatim — never regenerates the shell. An
artifact write that fails must not be reported as success.

**C10 · versioning-routing** — One live version of a deliverable per working folder,
`{YYYY-MM-DD}_{Description}.ext`, no `v#` suffixes. Superseded **minor** (presentation
changed) → delete, human-gated; **major** (substance/source changed, an approved checkpoint,
or anything sent/presented externally) → that folder's `Archive/`. Cleanup proposals are
always a specific keep/drop list.

**C11 · structured-options** — Every question to the user defaults to structured numbered
options — including info-gathering asks reframed as choices — recommended default first,
each option a complete outcome, an escape hatch, ≤4 options, one decision per question.
Inline prose questions only when an ask genuinely cannot be optioned.

**C12 · description-limit** — A skill's frontmatter `description` is ≤1024 characters,
verified mechanically on every change (CI enforces).

**C13 · presence-is-not-capability** — Detect what's connected and degrade gracefully;
never hard-fail on a missing tool. A configured integration is probed before first use each
session; a check that cannot run reports a loud SKIP, and a summary containing a SKIP never
reads as green.

**C14 · render-before-gate** — "As shown" refers ONLY to content rendered in the SAME
assistant turn that fires the gate question: render the content, then invoke the
structured-question tool in that same turn, option labels carrying the items' stable
identifiers (state ids like `appr-YYYYMMDD-NN`, task ids, opp numbers) verbatim — never
re-summarized nicknames. A render in any earlier message of this chat, however recent,
does not count; every revision and every chained/auto-invoked path re-renders in full
before asking again. "Rendered" means, by operation: file/artifact writes — the exact
text being written; state mutations — every user-meaningful item as `{id} + full
proposed action` (before → after for rewrites); delete/move — the exact source,
destination, and affected item. Destructive operations (file delete/move) are decided
per item, never wholesale, even when displayed inside a consolidated pass. Self-check,
answered audibly in the gate turn itself ("gating {N} items, all rendered above"): is
everything this approval writes rendered in THIS turn, at its required fidelity? Any
"no" → render now, then ask.
